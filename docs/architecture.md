# Architecture

> Keep this file current. If a decision is made during a session, update it before ending the session.

---

## Overview

This repo is a template for a skills-based AI collaboration pipeline. It defines the architect/executor split, the skills system, and cross-repo knowledge transfer patterns. Use it as a starting point for any new project, or run `/bootstrap` to migrate an existing project into it.

---

## Tech Stack

- No code dependencies — pure markdown and conventions
- Compatible with Claude Code, Windsurf, Cursor, and any AI tool that reads markdown

---

## Project Structure

```
pipeline/
  skills/                        # Canonical pipeline skill files — edit here to update the pipeline
    architect.md                 # Full architect protocol
    executor.md                  # Full executor protocol
    generate-consumer-skill.md   # Cross-repo skill generation protocol
    create-migration.md          # Cross-repo migration protocol
    bootstrap.md                 # Repo setup and migration protocol

.claude/
  commands/                      # Claude Code slash commands (thin wrappers → pipeline/skills/)

.windsurf/
  skills/                        # Windsurf skills (thin wrappers with frontmatter → pipeline/skills/)
    architect/
    executor/
    generate-consumer-skill/
    create-migration/
    bootstrap/

docs/
  architecture.md                # This file — system source of truth
  sync-queue.md                  # Pending doc updates (executor writes, architect reads and clears)
  context/                       # Domain-specific context files (architect-managed)
  migrations/                    # Cross-repo migration tasks (architect creates, manually carried over)

tasks/
  _template.md                   # Task template
  done/                          # Completed tasks

skills/
  exports/                       # Consumer skills generated by /generate-consumer-skill

CLAUDE.md                        # Thin bootstrap: loads architect protocol
AGENTS.md                        # Thin bootstrap: loads executor protocol (or architect in architect mode)
```

---

## Key Architectural Decisions

**Skills as the unit of protocol**: Pipeline rules and cross-repo knowledge are stored as skill files rather than embedded in CLAUDE.md/AGENTS.md. This keeps bootstrap files permanently thin, makes the protocol independently updatable, and makes knowledge portable across repos and tools.

**Single canonical source**: `pipeline/skills/` is the only place to edit protocol content. Tool-specific files (`.claude/commands/`, `.windsurf/skills/`) are thin wrappers that load the canonical files. When you update the protocol, update `pipeline/skills/` and copy the content to tool wrappers.

**Explicit context assignment**: Tasks explicitly list the context files they need. The executor does not scan the repo — it reads only what the task specifies. This keeps execution predictable and context windows lean.

**Cross-repo via skills**: Knowledge that needs to travel to other repos is packaged as a consumer skill in `skills/exports/`. Changes that need to propagate to dependency repos are packaged as migration files in `docs/migrations/`. Neither mechanism requires live access to another repo.

## Modules & Domains

**Pipeline Core** (`pipeline/skills/`)
The architect and executor protocols. Stable — only change when improving the pipeline itself, not for project-specific needs.

**Cross-Repo Layer** (`generate-consumer-skill`, `create-migration`)
Mechanisms for producing knowledge artifacts that travel to other repos. Consumer skills describe the public surface of this repo. Migration files describe changes that belong in a dependency.

**Bootstrap** (`bootstrap.md`)
Handles both new repo initialization and migrating existing repos into this pipeline. Tool-aware — generates the right config files for Claude Code, Windsurf, Cursor, or others.

---

## External Dependencies & Integrations

None. This pipeline has no external dependencies by design — it is pure markdown and conventions that any AI tool can read.

---

## Known Constraints & Non-Negotiables

- Only the architect creates or modifies `docs/context/`, `docs/architecture.md`, `AGENTS.md`, and anything in `pipeline/`
- The executor never modifies pipeline or documentation files directly — only flags via `docs/sync-queue.md`
- Tasks must be self-contained: the executor should never need to scan the repo to complete a task
- Consumer skills in `skills/exports/` must be self-contained: consuming repos should not need to open this repo to use them
